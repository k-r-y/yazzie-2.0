State Machine Audit ("Orphaned States")

Finding: The application code expects to rely on a pending status, but the database schema strictly defines: booking_status ENUM('inquiry','confirmed','completed','cancelled'). If the app attempts to insert pending (as seen in bookings.php:314), the database will reject the transaction in strict mode.
Actionable Fix: Unify the data layer to accommodate this middle state. 
-- Apply this schema adjustment
ALTER TABLE bookings MODIFY COLUMN booking_status 
ENUM('inquiry', 'pending', 'confirmed', 'completed', 'cancelled') NOT NULL DEFAULT 'inquiry';

3. Bug & Vulnerability Detection

Logic Bombs

The "Downpayment" Gap: In bookings.php, standard floating-point decimals are compared with basic subtraction arithmetic ($downpayment < $minDP - 0.01). This is fundamentally fragile due to IEEE 754 float precision in PHP leading to rounding discrepancies during edge totals.

Actionable Fix: Convert all financial records mathematically utilizing minor currency units (cents via integer logic / bcmath), or cast amounts explicitly.

Concurrency & Deadlocks

In bookings.php:

$avail = $pdo->prepare("SELECT COUNT(*) FROM bookings WHERE event_date = :date FOR UPDATE");
Vulnerability: InnoDB row-level locking (FOR UPDATE) on a COUNT(*) aggregation does not lock non-existent rows. Two concurrent requests requesting the exact same empty target date will both read 0 and wildly insert, violating your single-booking constraint.
Actionable Fix: Enforce unique dates directly against the Database schema level, relegating InnoDB to orchestrate the race condition atomically via collision errors. 
ALTER TABLE bookings ADD UNIQUE INDEX idx_unique_event_date (event_date);

4. The "Extreme" Edge Case Suite

  • Data Extremes: Supplying a 10,000-character string payload to event_location pushes system memory to pure TEXT limits. Max boundaries utilizing strlen() on string structures must be enforced before querying the DB. (Kudos on explicitly guarding maximum 300 Pax bounds, however).
  • Temporal Issues: PHP inherently manipulates through Asia/Manila while DB sessions natively hardcode +08:00. While synonymous, crossing midnights during extensive processing can invoke disjointed timestamps between instantaneous PHP variable checks and server-sided CURRENT_TIMESTAMP triggers during latency drift.
  • Partial Failures: The Phantom Commit. sendBookingConfirmation() is purposefully commanded after $pdo->commit(). If Google's SMTP drops the TCP connection off a spike, the app crashes via an uncaught Exception, outputting a 500 error to the browser. But wait! The SQL row is already persisted! The client believes they failed, while your kitchen believes they're confirmed.
Actionable Fix: Encase post-commit dispatchers with localized fault tolerance. 
try {
    sendBookingConfirmation([...]);
} catch (\Exception $e) {
    error_log("Non-fatal: Email dispatch failed for booking ID " . $newId);
    // Suppress error and proceed to issue the valid HTTP 201 Created JSON Response
}
